# StatsTid Roadmap

> Technology stack, phased milestones, and detailed next-sprint planning (rolling detail). See [SYSTEM_TARGET.md](SYSTEM_TARGET.md) for product definition, [CLAUDE.md](CLAUDE.md) for governance.

## Technology Stack

- **Backend**: C# / .NET 8 (Minimal APIs)
- **Frontend**: React 18 + TypeScript + Vite + ShadCN/ui (Radix primitives for complex interactions) + CSS Modules + IBM Plex Sans. Visual language inspired by Det F√¶lles Designsystem (designsystem.dk). See ADR-011.
- **Event Store**: PostgreSQL with custom event tables (via Npgsql, no EF Core)
- **Containerization**: Docker Compose (8 services)
- **Testing**: xUnit
- **Serialization**: System.Text.Json with polymorphic type handling
- **Architecture**: Event sourcing, outbox pattern, CQRS-lite
- **Rule Engine**: Pure functions, no I/O, deterministic, version-aware (OK24+)

## Deployment Model (decided 2026-05-18)

- **Single logical deployment**, single PostgreSQL database, single application instance. NOT per-customer deployment (interpretation 3 in the multi-tenant fork analysis).
- **Target tenant count at launch: ~150 institutions** (Danish state agencies / ministries / institutions that share the same overenskomster).
- **Tenants map to top-level organizations** in the materialized-path org hierarchy (ADR-008). No separate `tenant_id` abstraction today ‚Äî institution = top-level org node, with intra-institution org structure underneath. **ADR-025 D8 (ACCEPTED S38, 2026-05-21)** introduces an explicit `institutions` table (1:1 with top-level org via `org_id`) for billing/contract/onboarding affordances; the materialized-path "institution = top-level org" convention is preserved.
- **Agreements are GLOBAL across tenants** ‚Äî `agreement_configs` rows (AC/HK/PROSA/AC_RESEARCH/AC_TEACHING + OK versions) are a single registry maintained centrally. When the national overenskomst updates, one seed change propagates to all 150 institutions. No tenant-keying on agreement_configs.
- **Per-institution variations live in `local_configurations`** per ADR-017 ‚Äî keyed by `org_id` (institution's top-level org). Local overrides operate WITHIN the global agreement framework.
- **Within-agreement roles (fuldm√¶gtig / specialkonsulent / chefkonsulent) are also GLOBAL** by the same logic ‚Äî defined nationally by the overenskomst, not per-institution. The pending ADR-024 (role-within-agreement modeling) places the role registry at the same architectural layer as `agreement_configs`.
- **Glocal rule encoding principle (decided 2026-05-18)**: rule *interpretation* is GLOBAL ‚Äî one canonical encoding per cell, applies to all 150 institutions; rule *parameters that the overenskomst explicitly delegates to local agreement* are LOCAL ‚Äî vary via `local_configurations` per ADR-017. **No per-institution interpretation override is permitted** (ADR-026 candidate REJECTED before being filed). Product rationale: StatsTid acts as a standardization mechanism ‚Äî adopting it means joining a community of institutions that interpret the rules the same way. The system's encoded interpretation becomes a forcing function on the customer/employer-side negotiation. Strategic trade-off accepted: shipping cadence on disputed cells couples to negotiation cadence. Disputed cells block sprint dispatch until designated decider resolves. Default interpretation authority for launch: employer-side (Personalestyrelsen / Medst circul√¶r); deviations documented explicitly with decider + date.
- **Rule correction policy: supersession-by-default, bug-correction-when-classified (decided 2026-05-18)**: discovered discrepancies are classified as either (a) **interpretation change** ‚Äî the parties agree on a new reading or amend the overenskomst; encoded rule was correct at the time ‚Üí **supersession from effective date forward**, no retroactive recomputation, past periods replay against prior interpretation; OR (b) **bug** ‚Äî the encoding never matched the agreed interpretation ‚Üí **retroactive correction permitted** via in-band recompute. Classification is a governance step (encoding owner + product owner review) performed BEFORE any code change. The AC=UDBETALING case discovered 2026-05-18 is classified as bug-with-no-past-impact (pre-launch, no past periods exist). **Replay determinism preserved in both cases**: bug corrections produce a NEW segment manifest with corrected result; the original manifest persists and remains replayable byte-identically. "Current truth for period P" = latest manifest's result; "historical truth" = the manifest that existed at any prior point in time. **No per-institution opt-in/out**: bug corrections apply globally to all 150 institutions or to none (operator decision, not customer-side choice ‚Äî preserves glocal principle). Severe bugs that the in-band correction can't represent (e.g., bug in recompute logic itself, SLS-rejected batch) remain handled out-of-band. ADR-024 commits to the supersession definition + classification governance; **ADR-027 candidate (post-launch)** scopes the bug-correction workflow + event type + SLS reconciliation pattern. Pre-launch posture: rule corrections are seed edits with no replay-determinism cost ‚Äî this is the **last free-correction window**, which raises the stakes on Phase A audit thoroughness.
- **Cross-tenant data isolation** is enforced by `OrgScopeValidator` (ADR-009) + JWT scope-binding + the materialized-path scope check. LocalAdmin scope = institution subtree; GlobalAdmin = SaaS operator only (no per-institution user gets GlobalAdmin).
- **Out-of-scope at launch** (deferred to Phase 5+ "agreement expansion" backlog): police, military, technical, healthcare, vocational-teacher, court-staff, church-employed agreements. Adding new agreements later is additive (no schema change required) ‚Äî only seed rows + wage-type mappings + test fixtures.

#### Phase 4c.6 ‚Äî Original entry (preserved for historical context ‚Äî S26 Step 7a B1+B2+cycle-2-P1 carry-forward)

Root cause: S22's `ConfigEndpoints` exemplar succeeded because reads come from the `local_agreement_profiles` projection table; for `SkemaEndpoints` / `TimeEndpoints` / `BalanceEndpoints` / `ComplianceEndpoints`, **there is no projection table ‚Äî reads come straight from `events` via `IEventStore.ReadStreamAsync`**. **Phase 4c.6 also fixes a pre-existing data-consistency bug surfaced by S26 Step 7a cycle 2** at `SkemaEndpoints.cs:414-418` (the `if (!success) continue;` branch). Phase 4c.6's projection-table redesign supports BOTH atomic-tx semantics AND synchronous read-your-write ‚Äî the only correct resolution for the trichotomy.
