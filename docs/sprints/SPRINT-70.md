## Pinned rules (Step-0b cycle-1+2 absorptions — these are the contract; agents implement them verbatim)

**R1 — Deactivation lifecycle, provenance-tracked.** `is_active` has a pre-existing second writer (the AdminEndpoints user PUT, `AdminEndpoints.cs` ~L1020-1042; the soft-delete semantic is documented at `UserRepository.cs:94-97`), so the lifecycle must never blindly flip it back. New column `users.end_date_deactivated BOOLEAN NOT NULL DEFAULT FALSE` (provenance). Semantics: `employment_end_date` = the LAST day employed; the lifecycle deactivates when Copenhagen business date > `employment_end_date`. (a) Setting an end date already passed → same-tx `is_active=false` + `end_date_deactivated=true`. (b) Future-dated → no flip now; the poller flips when the date passes (R2). (c) Clear → reactivate ONLY when `end_date_deactivated=true` (then reset it); clearing an end date on a manually-deactivated user clears the date but does NOT reactivate. (d) Setting an end date on an already-manually-inactive user records the fact but does NOT claim provenance. (e) Every lifecycle flip (same-tx and poller) reuses the existing user-deactivation side-effect path (`ReportingLineManagerDeactivated` emission per `ReportingLineRepository.cs:111`). (f) DELIBERATE coherence point (record in code comment): the admin general PUT filters `is_active=TRUE`, so it can neither edit nor reactivate a deactivated user — R1(c) is the ONLY reactivation path for lifecycle-deactivated leavers.

**R2 — The deactivation flip is NOT D13-gated; settlement is. Step A is atomic and version-bumping.** `SettlementCloseService` restructures so **Step A runs BEFORE the D13 gate check** (the gate's early return at `SettlementCloseService.cs:189` must not skip it): **Step A (UNGATED)** — **per leaver, under that leaver's R12 lock** (NOT a set-based bulk UPDATE — the per-employee lock + per-employee event payload force a loop: lock → single-row UPDATE → side effects + event → commit), ONE predicate-guarded UPDATE (`employment_end_date IS NOT NULL AND employment_end_date < copenhagen_today AND is_active = TRUE`, full predicate re-evaluated in the UPDATE's WHERE) that sets `is_active=false`, `end_date_deactivated=true`, AND **increments `users.version`** (ADR-018 D7 — a held ETag must not survive the transition), with R1(e) side effects + the `EmployeeEndDateDeactivationApplied` emission (version-before/after; system actor matching the existing settlement actor convention) in the SAME tx. **Step B (D13-gated, `Settlement:GoLiveDate`)** — settle leavers (R4); **the TERMINATION go-live anchor is the end date: Step B settles ONLY leavers whose `employment_end_date` falls strictly after `Settlement:GoLiveDate`** — earlier leavers are pre-launch boundaries the system never tracked and remain the manual fallback per D13 (pinned test). Pinned test: dormant settlement gate + passed end date ⇒ user still deactivated.

**R3 — Leaver selection is keyed on `employment_end_date`, never on bare `is_active=FALSE`.** Step B's predicate: `employment_end_date IS NOT NULL AND employment_end_date < copenhagen_today` and no active (non-REVERSED) settlement of **ANY trigger** for the end-date ferieår tuple (the R8 any-trigger anti-join — a TERMINATION-only anti-join would leave an R7b-conflicted leaver in the due set forever, re-firing the refusal signal every 5-min poll against D10's flagged-ONCE idempotency; R7b stays the in-tx race backstop only). A manually-deactivated user with no end date is NEVER settled as a leaver.

**R4 — A leaver's ENTIRE unsettled tail surfaces via the state machine's OWN semantics; nothing silently vanishes (D5+D10).** The YEAR_END due-enumeration's `is_active = TRUE` filter (`SettlementCloseService.cs:303`) is widened to ALSO include leavers (`is_active = FALSE AND employment_end_date IS NOT NULL AND employment_end_date < copenhagen_today` — future-dated and manually-inactive users excluded), **capped at ferieår ≤ the end-date ferieår** (no post-termination years are ever generated); post-go-live boundary filter unchanged. For a leaver: the **end-date ferieår** → TERMINATION crystallization (R5) via Step B. **Every OTHER due ferieår** (e.g. the prior year whose boundary passes after deactivation) → a `trigger=YEAR_END` row written **fail-closed PENDING_REVIEW** — NO auto-partition (no §21/§24 arithmetic invented for leavers). Row shape per the **S68 flag convention** (the flagged remainder is stamped on `forfeit_days` — a FLAG, not a disposition; the resolve endpoint's override guard and the D9 reader are built on it, `VacationSettlementEndpoints.cs` ~L449-469, `BalanceEndpoints.cs` ~L316-331): `transfer_days = payout_days = 0`, **`forfeit_days = full disposable`**, snapshot captured at the SAME strictly-dated ferieår-start anchors as R5 **with a `DeferredDisposition` marker** distinguishing no-partition from a computed partition (D3 replay determinism + the data for the deferred owner ruling). Resolved via the EXISTING CAS manual FORFEIT/DEFER workflow UNCHANGED (FORFEIT resolves the flagged quantity; **DEFER is pinned marker-only in 3a** — it writes NO carryover). Invariant + test replacing the former DEFER-guard: **no resolve disposition writes `carryover_in` in 3a** (the carryover-into-settled-year rejection rule belongs to 3b/slice-4, where a carryover-writing resolution first exists). **Two leak-proofing pins (cycle 4):** (a) the enumeration's ACTIVE branch excludes `employment_end_date IS NOT NULL AND employment_end_date < copenhagen_today` — a leaver whose Step-A flip failed (per-leaver isolation: log + continue) must NEVER traverse the normal §21/§24 auto-partition (which would write carryover + emit `VacationAutoPaidOut` into the S69 §24 staging); those rows belong exclusively to Step-A retry + the leaver branch; (b) inside `SettleAsync`, the leaver/no-partition decision keys on the in-lock re-read `employment_end_date` (R3's principle applied in-service), so a flip race can never yield a partitioned settlement for a terminated employee.
