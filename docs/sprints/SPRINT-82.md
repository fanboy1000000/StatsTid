- **Routes** (`frontend/src/App.tsx`): `/login`→LoginPage; index → `/tid/registrering` (SkemaPage); `/tid/mine-perioder` (MyPeriods — employee submit surface); `/godkend/godkendelser` (ApprovalDashboard, `RequireRole minRole=LocalLeader`); employee/leader/HR/admin role-gated subtrees.
- **The stack**: docker-compose 7 services (`docker/docker-compose.yml`, `:5100-5700`), seeded init.sql; dev users share pw `password`; the seeded reporting edge `emp001 → mgr03` is in tree STY01 (init.sql) — so `mgr03` is the natural approver for `emp001`'s period. `hr01` is STY02 (no seeded report — NOT used for the approval journey).
- **Serve/proxy**: the `/api`→`:5100` proxy lives in `frontend/vite.config.ts` `server.proxy` (DEV-server only; `vite preview` has no proxy) → the harness MUST run against the vite **dev** server.
- **CI precedents**: the `smoke-tests` job (`.github/workflows/ci.yml`) is self-contained (own checkout + `docker compose up -d --build` + host-side `/health` loop on 5100-5700 because container healthchecks lack curl) and DELIBERATELY independent of `build-and-test` (S63: "one red must not blind another"); the `frontend-build` job gives the `setup-node` + `npm ci` pattern.
- **Approval state is NOT seeded** (both review lenses): init.sql has no `INSERT INTO approval_periods`; the pending views read only SUBMITTED/EMPLOYEE_APPROVED (`ApprovalPeriodRepository.cs:82/:206`); the submit endpoint creates periods (`ApprovalEndpoints.cs:78/:130`); approve/reject are terminal. So the approval E2E MUST create its own SUBMITTED period(s) via the employee submit flow.
- **Real clock**: the running stack uses `TimeProvider.System` (`Program.cs:295`) + approval "today" reads `DateTime.UtcNow` (`ApprovalPeriodRepository.cs:228`) — NO FixedTimeProvider (unlike the vitest tier). Date-fragile flows flaked S77/S78; FAIL-002 = Docker sheds containers under churn.
- **The a11y target**: `frontend/src/pages/approval/ApprovalDashboard.tsx` has TWO hand-rolled plain-div dialogs (the rejection dialog `~:381` + the enforcement-confirmation dialog `~:417`) — `styles.dialogOverlay`/`styles.dialog`, manual click-stop, NO focus-trap/Escape/aria.

## Pinned requirements
- **R1 (harness scaffold).** Add `@playwright/test` (frontend devDep) + `frontend/playwright.config.ts` (chromium, headless, `retries:2`, `trace:'on-first-retry'` + video-on-retry, `webServer` auto-starting `npm run dev`) + a `frontend/e2e/` spec dir + an `npm run e2e` script. **`baseURL` is parameterized** via env (default `http://localhost:3000`) so the same journeys can later target a deployed/staging environment (the 2026-05-18 deployment model) — NOT CI-only-shaped. Kept entirely separate from vitest (`src/**/*.test.tsx`).
- **R2 (3 critical journeys — self-state-creating, re-run-tolerant).** (a) **login** as a seeded user → lands on `/tid/registrering` (the JWT auth round-trip); (b) **Skema registration**: `emp001` registers an absence on `/tid/registrering` → persists → visible after reload; (c) **approval**: create the SUBMITTED period via the **MyPeriods submit form** (`/tid/mine-perioder` → `POST /api/approval/submit`, which creates a raw SUBMITTED period directly — NO Skema-content dependency, `ApprovalEndpoints.cs:187`; do NOT route through Skema's `submitAndApprove`, which immediately employee-approves and requires full workday coverage — both review lenses). Then **`mgr03`** (the seeded `emp001→mgr03` STY01 edge) **approves** it; a SECOND, isolated SUBMITTED period is **rejected** via the reject dialog. TWO separate periods because approve/reject are both terminal. `mgr03` for BOTH; NOT `hr01` (STY02, no seeded report). **Assertion target pinned: raw SUBMITTED** (manager-visible) → APPROVED / REJECTED (not EMPLOYEE_APPROVED). **Auth = fresh login per spec** (isolation-first; a shared `storageState` is a recorded perf follow-on). **Each mutating spec owns runtime-unique state keyed off a per-run nonce** (a timestamp-derived unique month, NOT just "this month" — so a re-run after a mid-spec failure can't 409 on the `/submit` existing-SUBMITTED guard `ApprovalEndpoints.cs:109` nor go green-but-weak against the persistent local volume).
- **R3 (gated CI `e2e-tests` job).** A NEW dedicated job (independent of `build-and-test` + `smoke-tests`): checkout → `setup-node` → **`npm ci`** (frontend) → `docker compose up -d --build` → host-side `/health` wait (5100-5700, the smoke pattern) → `npx playwright install --with-deps chromium` → `npm run e2e` (the playwright `webServer` starts the FE) → **GATING** (a real failure blocks). On failure (`if: failure()`): capture `docker compose logs --no-color` + upload it alongside the Playwright HTML report + traces (smoke-job parity — an E2E failure rooted in a backend-service crash is undiagnosable from the Playwright trace alone, internal-lens W). `docker compose down -v` teardown `if: always()`. **Trigger: master push + PR** (matching the suite — master-only would weaken the claimed gate, Codex); chromium-only headless, no browser matrix (CI-cost discipline on the metered $20 Actions budget).
- **R4 (date determinism against the REAL clock).** Specs compute the target month/day at RUNTIME in UTC and pick a **non-boundary** day (mid-month, a weekday) — never a hardcoded calendar date, never a month-end/year-end/weekend edge (the S77/S78 rollover flake class). Helper centralizes the date derivation.
- **R5 (a11y retrofit — BOTH dialogs).** Migrate BOTH `ApprovalDashboard.tsx` plain-div dialogs (rejection + enforcement-confirmation) to the kit accessible `Dialog` (`components/ui/Dialog.tsx`; Radix focus-trap/Escape/`aria`), behavior-preserving (the reject-reason input + the confirm/enforce actions unchanged). **EXTEND the existing** `frontend/src/pages/approval/__tests__/ApprovalDashboard.test.tsx` (it ALREADY exists, with the recorded dialog-a11y follow-up placeholder at `~:531-537` — Codex corrected the plan's stale "net-new" assumption; do NOT plan from false ground truth). The kit `Dialog` is controlled via `open`/`onOpenChange` — the migration MUST map `open={rejectTarget !== null}` / `open={enforcementDialog !== null}` AND route Radix's `onOpenChange(false)` (Escape / overlay-click / the built-in close-X the plain-divs lack) to the existing `closeRejectDialog`/`closeEnforcementDialog` handlers, else Escape closes visually but leaves stale React state (internal-lens W). Tests assert: role=dialog + focus-trap; **Escape closes WITHOUT firing approve/reject**; the confirm/enforce buttons still POST the SAME payloads (the reject-reason `trim()`-gated submit + the enforce action unchanged); the textarea `autoFocus` vs Radix initial-focus interplay holds.
- **R6 (grade = A− with named residual; docs).** `docs/QUALITY.md` updated: FE **B→A−**, with **visual-regression + component-docs** explicitly named as the deferred residual (NO flat-A claim — the S77 lesson). `docs/FRONTEND.md` documents the E2E harness (what it covers + how to run locally + the CI job). The grade lift is honestly supported by a genuine 3-journey gated harness.
- **R7 (invariants / scope).** FE + CI-config ONLY — no backend/API/schema/event change. The existing 459 vitest tests stay green (±the a11y component change); the E2E suite is additive. Nothing above priority #9 is touched.
