## Premise
The codebase resolves the entitlement-config row that governs a ferie(år)'s quota (`annual_quota`/`carryover_max`) via **two competing temporal-anchor conventions**, and the write/validation + balance-summary paths disagree with the newer year-overview path. For an employee whose OK version OR agreement differs between the entitlement-year-start and the operation date, quota *validation* (and the balance *summary*) resolve quota against a **misaligned anchor triple** — live OK version + month-start agreement + year-start row-date — whereas the year-overview resolves the **year-start-aligned** triple. Result: a validation-vs-display/accrual split-brain (an employee can be blocked or allowed against quota numbers that don't match their own year balance). This sweep unifies the 3 misaligned sites onto the year-start-aligned convention. Priority #4 (version correctness).

## Source-verified ground truth (cited)
- **The two conventions:**
  - **Convention A — misaligned triple:** live `user.OkVersion` (today) + **month-start-dated** `agreementCode` (`= pastEffectiveAgreementCode ?? user.AgreementCode`, `SkemaEndpoints.cs:596-598`, `BalanceEndpoints.cs:96-98`/`:474-476`) + a **year-start** config-row date. Sites: `SkemaEndpoints.cs:1094-1096` (quota validation, absence-registration write path — a PRE-transaction validation, runs before `BeginTransactionAsync` `:1315`, so the non-tx `GetByTypeAtAsync` overload suffices), `BalanceEndpoints.cs:227-229` (`/summary`), `BalanceEndpoints.cs:510-512` (`/series`).
  - **Convention B — year-start-aligned:** `BalanceEndpoints.cs:833-844` (`ResolveDatedConfigAsync`) + `:955` (YearOverview): dated agreement (`ResolveAgreementAtAsync(ferieaarStart)`, `:790`/`:836-838`) → dated OK (`OkVersionResolver.ResolveVersion(ferieaarStart)`) → `GetByTypeAtAsync(..., dated-OK, ferieaarStart)`, with a graceful live fallback (ADR-023 D3). The S65-hardened path; matches ADR-020/021's dated-config intent.
- **ADR-021 D4/D5 (the binding constraints):** D4 two-step consumption (live read → derive year-start from `ResetMonth` → dated read at year-start). D5 freezes `reset_month`/`accrual_model` **within a natural key** `(type, agreement, ok)` via the admin-CRUD 422 guard (`EntitlementConfigEndpoints.cs:447-461`) — this is WHY Convention A is internally consistent today (Step 1 and Step 2 read the SAME key). The sweep moves Step 2 to a DIFFERENT (year-start-dated) key, so the per-key freeze no longer bridges Step 1↔Step 2.
- **`reset_month` is TYPE-uniform across agreements in the SEED, but NOT schema-enforced (Step-0b Codex BLOCKER):** `DefaultEntitlementConfigs.cs` + `init.sql:1532-1605` seed the same `reset_month` per TYPE across ALL agreements (VACATION=9, SPECIAL_HOLIDAY=1 [S80, accrual 1-Jan-anchored independent of reset_month], CARE_DAY/CHILD_SICK/SENIOR_DAY=1) — legally grounded (the Danish ferieår is national, not per-agreement). BUT the normal admin create path (`EntitlementConfigEndpoints.cs:174-180`, `AgreementEntitlementEndpoints.cs:149-160`) can mint a fresh natural key with a divergent `reset_month` (D5 freezes only EDITS within an existing key, not new-key CREATES). **Precision (both cycle-2 lenses):** VACATION is ADDITIONALLY schema-pinned to `reset_month=9` (DB CHECK `entitlement_configs_vacation_reset_month` `init.sql:1393-1395` + the create 422-guard `EntitlementConfigEndpoints.cs:179-186`), so the divergence window is reachable ONLY for the unconstrained IMMEDIATE calendar types (CARE_DAY/CHILD_SICK/SENIOR_DAY) and agreement-scoped creates — NOT VACATION. So a real `(employee, type)` CAN reach divergent operation-date-key vs year-start-key reset months for those types after an agreement/OK change → the year-start derivation must NOT rely on the seed invariant. (This is why §"Design decision" chooses re-derivation, not a test-only guard; re-derivation is correct for the unconstrained types AND removes the dependency cleanly for VACATION too.)
- **Deliberate NON-targets:** `VacationSettlementService.cs:820` (accrual-start resolver — live fallbacks then a final fail-closed throw) and `:1415` (a HYBRID — a fail-CLOSED guard at `:1432-1440` for `terminationCutoff != null || deferredDisposition`, in FRONT of the verbatim D9 graceful chain at `:1441-1450` for the active-employee YEAR_END; ADR-033 D10). Excluded from the hoist because their **strictly-dated, never-graceful-for-leavers** posture differs from the shared graceful family — even though `:1415`'s active branch duplicates the graceful chain. A regression pins the leaver path stays fail-closed.
- **Deliberate live-version asymmetries (out of scope, documented):** `SkemaEndpoints.cs:1510` (S66/TASK-6603 `TimeEntryRegistered` live-OkVersion), `:912`/`:1966` (`SENIOR_DAY` S59, flagged `:1988-1992`), display/"today" reads (`BalanceEndpoints.cs:131`/`:692`).

## Design decision — the ResetMonth cross-key fork (Step-0b Codex BLOCKER), RESOLVED via RE-DERIVATION
**Resolution: derive the entitlement-year-start from the config governing the OPERATION DATE (dated OK + dated agreement at the absence/month date), then read the Step-2 quota config at that year-start (dated OK + dated agreement at the year-start).** This is correct-by-construction regardless of cross-key `reset_month` divergence, and introduces NO chicken/egg: the operation date is known a priori (the absence date in Skema / the requested month-start in Balance), so dating the OK version (`OkVersionResolver.ResolveVersion(opDate)`) + agreement (`GetByUserIdAtAsync(opDate)`) for the Step-1 `reset_month` read needs no config first. The operation-date `reset_month` defines which entitlement year the operation consumes from (the defensible rule when an agreement changed mid-year). We do NOT rely on the seed-level type-uniform `reset_month` assumption (Codex BLOCKER: a normal admin create can break it). The alternative — keep the live-key Step-1 + a test-only guard — was REJECTED (a test does not enforce production correctness); the other alternative — admin-write-path enforcement of type-uniform `reset_month` — was REJECTED as out-of-scope (admin-write territory) and legally over-constraining. R5 keeps a discriminating test as defense-in-depth, but correctness no longer DEPENDS on it.

- **R2 (Step-1 year-start derivation — re-derive from the operation-date key).** Step-1's `reset_month` read is anchored to the OPERATION DATE (dated OK + dated agreement at the absence/month date), not the live key, so the derived year-start is correct even when `reset_month` diverges across keys. - **R4 (graceful-vs-fail-closed isolation — HARD).** The shared resolver is the GRACEFUL family only. - **R5 (the `reset_month` divergence test — defense-in-depth).** A discriminating test exercises an employee+type whose operation-date-key and year-start-key `reset_month` DIFFER, proving the re-derived year-start (R2) is correct.
