## Pinned requirements
- **R1 (7800 — approval in-lock edge-auth, Leader-arm).** In approve / reject / reopen-Leader-arm: begin the tx (READ COMMITTED), **acquire the period's employee tree advisory via the shared drift-guarded acquire (R9)** as the FIRST LOCK-BEARING statement (R9 handles the derive → acquire → drift-revalidate → re-acquire-on-drift, so the approve path holds the CURRENT non-stale key). **Then re-evaluate the designated-edge authorization STRICTLY AFTER the advisory is acquired.** The serialization comes from the action tx HOLDING the `reporting-tree` advisory: a concurrent in-scope revoke (reporting-line remove / admin-vikar) calling the same advisory BLOCKS *before* its mutation, so it cannot COMMIT while the action holds the lock — the committed edge state is therefore frozen for the action's duration. **The resolver re-read may run on its existing OWN-connection path (NO new `(conn,tx)` overloads required) — what matters is that it executes after the advisory is held, observing the frozen state** (Codex/Reviewer: the in-tx-overload design ALSO works but is more code + introduces an approve-path user-row `FOR UPDATE` deadlock surface; the post-lock-re-read design is simpler, equally serialized, and avoids that surface — chosen). This is TRUE serialization, NOT the window-tighten the owner rejected (a fresh-connection re-read with NO held advisory would be window-tighten; here the advisory is held). `asOf` computed at action-time (closes the asOf-midnight item). The org-scope JWT half is NOT re-checked (can't be). **CRITICAL (Step-5a-verified):** the re-read happens strictly after advisory acquisition, AND every in-scope revoke mutator (reporting-line remove + admin-vikar) provably holds the SAME advisory key (the residual vectors per R4 do NOT — documented, not assumed serialized).
- **R2 (7800 — the status race, BOTH arms).** Make the status transition CONDITIONAL — `UPDATE … WHERE period_id=@id AND status IN (<allowed>)`, `0 rows → 409` — on ALL transitions including **BOTH reopen arms (employee + leader)** and approve + reject. **The conditional UPDATE must be the FIRST mutation in the tx, BEFORE every side-effect enqueue** (Codex WARNING — approve/reject today enqueue a `FallbackTraversalWarning` BEFORE the status UPDATE at `ApprovalEndpoints.cs:257`, with the audit insert + action outbox at `:273`/`:281` after; reorder so the `0 rows` case short-circuits to 409 BEFORE any FallbackTraversalWarning / audit / outbox is written) → the **loser path emits NO fallback warning, NO audit row, NO outbox event** (pinned by ordering, not left to rollback semantics). The **EMPLOYEE reopen arm gets the conditional UPDATE but NOT the edge-auth advisory lock** (it carries no designated-edge authority).
- **R3 (7801 — cross-transfer serialization, FULLY closed via R9).** In `PUT /api/admin/users/{userId}`, when `primary_org_id` changes: acquire the `reporting-tree` advisory for BOTH the OLD + NEW tree roots (id-ordered, deterministic) **BEFORE** the `GetByIdWithVersionAsync` users `FOR UPDATE` (`:1035`, advisory-before-rows), via the shared drift-guarded acquire (R9); then the existing `users` UPDATE. Owner ruled **OQ-DRIFT = SOUND** → the S74-7403 cross-transfer residual is FULLY closed (NOT just the common case): R9's shared drift-guarded acquire is applied to EVERY tree-keyed mutator so no two paths can hold different keys under a simultaneous transfer. Replaces the in-code deferral at `ReportingLineEndpoints.cs:91-104/:568-576/:1278-1306` with the implemented guard.
- **R9 (the SHARED drift-guarded acquire — the load-bearing primitive, owner OQ-DRIFT=SOUND).** Add ONE shared helper (e.g. `AcquireTreeLockForEmployeeAsync(conn, tx, employeeId)` extending `ReportingLineRepository.AcquireTreeLockAsync:600`): (1) derive the employee's current tree root (unlocked); (2) acquire the `reporting-tree-{root}` xact advisory; (3) **re-derive the root UNDER the held advisory**; if it DRIFTED (a transfer committed between the derive and the acquire) → **ROLLBACK the whole tx + RETRY the entire request** with a fresh READ COMMITTED tx (Codex c3 BLOCKER — `pg_advisory_xact_lock` releases ONLY at tx end, so a mid-tx release/re-acquire is impossible, and retaining OLD while acquiring NEW invites lock-accumulation/A↔B deadlock; do NOT switch to session `pg_advisory_lock`/`unlock` — explicit-unlock/cancellation/pooling failure modes expand). The drift-check + retry happen BEFORE any user-row `FOR UPDATE` or mutation (so no side effects on the retried tx); **bounded ≤3 retries → a PINNED 409/503** (not an incidental 5xx). Applies to the **employee-current-root** mutators: reporting-line **assign** (`:81`), **remove** (`:275/:1231`), **acting** (`:558`), the **transfer** (R3, old+new roots), and the **approve** path (R1) — these all key on the same employee-current derivation → genuine mutual exclusion under a simultaneous transfer. **EXCEPTION (Codex c3 BLOCKER 2): the admin-vikar REVOKE path is NOT re-keyed** — it deliberately keys on the **PERSISTED** vikar root (`tree_root_org_id`, `:2448/:2468`) for revoke-safety (it must work for an INACTIVE manager whose current root can't be derived). R9 leaves it as-is; the approve-vs-vikar-revoke serialization holds when the persisted root == the employee-current root (the common case) and the post-transfer corner (persisted ≠ current) is a small NAMED residual (R4). A bounded-retry + no-deadlock + drift-rollback-retry test is mandatory (R6). **Preserve the existing S74-7403 five-cycle-crucible assign/delete tests** while adding the new R6 coverage.
