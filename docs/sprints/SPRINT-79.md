## Source-verified ground truth (dual-lens grounded, cited)
- **§22 (LBK 152/2024):** an employee prevented from holding earned vacation by an enumerated impediment (sickness/barsel) has up to **4 weeks** transferred automatically (no agreement, unlike §21) to the next afholdelsesperiode; if the impediment persists → §25 pay-out (DEFERRED). Crystallizes at the afholdelsesperiode boundary.
- **The carryover writer today** (`VacationSettlementService.cs:373`, `EntitlementBalanceRepository.cs:221`) is **§21-only, overwrite-not-add, skip-when-zero** — deliberately skip-when-zero to avoid clobbering the future §22 producer (the S68-W1/W3 decision). §22 term is hard-zero pre-slice-4 (`EntitlementBalanceRepository.cs:187`).
- **The `/resolve` endpoint** (`VacationSettlementEndpoints.cs:350,402`) is a single POST with a disposition discriminator ∈ {FORFEIT, DEFER, WAIVED}, MODREGNING parked, behind an ADR-019 CAS winner guard. **DEFER deliberately stays PENDING_REVIEW and writes no event/carryover** (`:539`) — so FERIEHINDRING is a NEW sibling branch, not a DEFER variant.
- **The events** `FeriehindringTransferred` (§22) + `FeriehindringPaidOut` (§25) exist as **define-only** classes, EventTypeMap-registered (`EventSerializer.cs:152`), round-trip-safe — but carry **no reason/provenance field** (identity + nullable snapshot + day-count only).
- **Reversal** (`SettlementReversalService.cs:186`) currently refuses only `transfer_days>0` carryover-writing rows (3b zero-bucket-only; carryover-compensating reversal is the named follow-up). The new §22 bucket must be reversal-aware.
- **Schema** (`init.sql:2752`): `vacation_settlements` has `transfer_days/payout_days/forfeit_days/review_disposition/claim_disposition_days`; `review_disposition` CHECK (`:2793`) excludes FERIEHINDRING.

## Pinned requirements
- **R1 (source-keyed carryover — closes S68-W3).** `carryover_in` is the additive §21+§22 sum, **composed in C# → exactly ONE `WriteCarryoverInAsync` overwrite** per settlement tx (never two writer calls). Skip only when the **combined** sum is zero. Replay-stable (pure function of snapshot + recorded resolution). Discriminating test: a settlement with BOTH §21 and §22 components writes the correct additive carryover; a §22-only settlement writes correctly without relying on skip-when-zero.
- **R2 (§22 rescues from §34 — separately-capped coexisting tranches; CORRECTED at Step-0b, both lenses BLOCKED the original "combined-ceiling/displacement" framing).** §21 and §22 do NOT compete for one ceiling — they are independent, coexisting transfer tranches (law-research `:16-19`; ferie-transfer-timing `:22-24`; Ferieloven §§21,22,24,34):
  - **§22 feriehindring** = up to **4 weeks (20 days)**, automatic (no agreement), sourced from the **impeded portion of the over-cap §34-forfeiture-candidate remainder** — i.e. §22 **RESCUES days from the §34 bucket** (it REDUCES the §34 forfeit; it does NOT displace §21).
  - **§21** = the **>4-week (5th-week) tranche** (≤ its existing `carryover_max`, ≤5 days), by written agreement, from the under-cap tranche (unchanged).
  - `carryover_in = transfer_days(§21) + feriehindring_transfer_days(§22)` — **NO combined `carryover_max` cap** on the sum (Codex BLOCKER 2: capping the combined sum at `carryover_max`/the 5th-week ceiling would wrongly cap a fully-feriehindret employee to 5 days instead of 20). Max same-year transfer = 25 days = §22(≤20) + §21(≤5).
  - **§34 forfeit = the over-cap §34-candidate remainder MINUS the §22-rescued days.** If a §21 quantity ever exceeds its valid >4-week tranche, the excess >4-week remainder lands in **§24 auto-payout, not §34** (law-research `:17-18`).
  - The ADR-033 **D6 "combined statutory transfer ceiling / §22 precedence" wording is over-modeled and MUST be amended** by this slice (7905). Discriminating test: §22 REDUCES the §34 forfeit (not displaces §21); a fully-feriehindret employee transfers up to 20 §22 days.
- **R3 (FERIEHINDRING disposition — explicit partial split).** A NEW terminal disposition on `POST /resolve` (sibling to FORFEIT/DEFER/WAIVED; NOT a DEFER variant). Body = impeded days (≤4wk §22) + reason. **The explicit partial split the current FORFEIT path refuses** (`VacationSettlementEndpoints.cs:669-675` — FORFEIT forfeits the WHOLE flagged remainder): bound **`0 < impeded ≤ min(current.ForfeitDays, 20)`** — BOTH the flagged §34-candidate remainder AND the independent §22 4-week/20-day statutory cap (Codex c2 W: `current.ForfeitDays` can exceed 20); `feriehindring_transfer_days := impeded`; **`forfeit_days := current.ForfeitDays − impeded`** (§22 transfer FIRST, §34 only the residual) — so the same day can never both transfer and forfeit. ONE atomic CAS tx: emit `FeriehindringTransferred` + the R1 source-keyed carryover overwrite (`transfer_days + feriehindring_transfer_days`) + `VacationForfeitedToFeriefond` for the residual `forfeit_days` (if >0) + `review_disposition='FERIEHINDRING'` + durable reason + version bump + `PENDING_REVIEW→SETTLED`. ADR-019 If-Match winner guard; concurrent completion → loser 409 (no double-emit). Reuse the existing CAS/advisory-lock resolve path (`:419-429,:525-535`); NO second mutation path to the row.
- **R4 (durable reason — event-primary).** The impediment reason persists durably with the **extended (not-yet-emitted, contract-safe) `FeriehindringTransferred` event field as the PRIMARY home** (replay-completeness — the audit rationale must reproduce from the event stream), **mirrored to a queryable `feriehindring_reason` column** on `vacation_settlements`. Audited; not merely API-accepted.
- **R5 (schema).** `feriehindring_transfer_days NUMERIC(6,2) NOT NULL DEFAULT 0` bucket + `feriehindring_reason TEXT` (R4); **`review_disposition` value-CHECK widened to include `'FERIEHINDRING'`** (DROP-IF-EXISTS + re-ADD named superset — S71 WAIVED precedent, no remediation); **AND the `vacation_settlements_disposition_state` CHECK widened to place FERIEHINDRING in the "resolves-the-review ⇒ NOT PENDING_REVIEW" group** (`init.sql:2803-2807` — rides SETTLED, survives REVERSED; without this the SETTLED resolve violates the existing state-coupling CHECK); a new `feriehindring_*`-paired CHECK (separate from `claim_disposition_paired`, which binds MODREGNING/WAIVED); non-neg CHECK extended to the new bucket; legacy ALTER block (all 3 upgrade paths); db-schema regen (`check_docs` green).
- **R6 (events/audit).** Wire `FeriehindringTransferred` **emit + the ADR-026 audit mapper + the DI registration** (alongside the existing settlement mappers at `Program.cs:234-240` — Codex W: the event is serializer-registered but has no mapper yet, so the audit half is currently missing) + the audit-projection-catalog doc entry (the catalog is test-parsed — SPRINT-66 lesson); round-trip/replay coverage green. `FeriehindringPaidOut` stays define-only (DEFERRED).
- **R7 (reversal bucket-awareness + accepted irreversibility).** The S71 reversal service treats `feriehindring_transfer_days>0` as a carryover-writing row — added to the SAME guard that refuses `transfer_days>0` (`SettlementReversalService.cs:186-195`), fail-closed/loud (the carryover-compensating reversal stays the deferred follow-up; no §22 compensating reversal in this slice). `SettlementReversed` + export-compensation bucket logic learns the new bucket; §24/§26 emitters confirmed unaffected; no silent mis-reversal. **Accepted limitation (recorded in R10 + the ADR amendment):** a FERIEHINDRING-resolved row is carryover-writing → SETTLED → **irreversible** until the compensating-reversal follow-up lands — a mis-recorded impediment count is not retractable in-slice (consistent with the 3b zero-bucket-only / manual-fallback posture; the operator entry is the legally-sensitive point of accountability).
- **R8 (§34 fail-closed preserved).** §34 is NEVER auto-forfeited for an unresolved impediment; the blind poll does NOT forfeit. A test pins that a DEFER'd / unresolved PENDING_REVIEW row is not auto-forfeited. §34 completes ONLY via the FERIEHINDRING/FORFEIT operator resolution.
- **R9 (partition integrity — full equation + correct enforcement).** The FULL partition reconciles (the equation must include the §24 + planned terms): `transfer(§21) + payout(§24) + feriehindring_transfer(§22) + forfeit(§34) + used + planned = disposable-basis` with NO overlap. **The `impeded ≤ current.ForfeitDays` bound is NOT a static `feriehindring_transfer_days ≤ forfeit_days` row CHECK** (Codex c2 BLOCKER — that wrongly rejects a FULL rescue: final `forfeit_days=0` while `feriehindring_transfer_days=20`). It is enforced two ways: **(a)** the resolve HANDLER reads the pre-resolution `forfeit_days` under the advisory lock and rejects `impeded > min(current.ForfeitDays, 20)` with a clean 422; **(b)** the residual `forfeit_days := current.ForfeitDays − impeded` is protected by the (extended) **non-negative-buckets CHECK** — an over-rescue drives `forfeit_days` negative → the CHECK rejects the tx (defense-in-depth, NOT the primary gate). §22-transfer and §25-payout are mutually exclusive (§25 absent this slice; the columns must not permit double-dispose).
- **R10 (non-goals — recorded follow-ups + the ADR amendment).** §25 `FeriehindringPaidOut` pay-out NOT implemented (manual/define-only). Blind poll-auto-§34 NOT implemented. The §22 carryover-compensating reversal NOT implemented (R7 fails closed → the FERIEHINDRING row is irreversible in-slice). **The ADR-033 D6 amendment** (correct the over-modeled "combined ceiling / §22 precedence" to the §22-rescues-from-§34 separately-capped-tranches model of R2) is REQUIRED, not optional. All recorded in the ADR-033 amendment (7905).
