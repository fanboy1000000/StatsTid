## Context

ADR-030 D7 deferred the vacation **settlement** chain — the period-end *execution* the S66 D9 disposition row only *displays*. Constraints (refinement fact-finding + verified law):

- **No salary/base-rate in-system** (`Amount = Hours × multiplier`; SLS applies salary). Settlement detects boundaries, executes balance/state mutations, and emits **day/hour-count wage-type lines** for SLS to monetize — it must NOT compute kroner.
- **Everything greenfield** (code-verified): no period-close trigger; `carryover_in` no non-zero production writer; no `employment_end_date`; no §21 agreement record; no settlement event/line.
- **Verified legal model** (§-spine, corrected S67): transfer-by-written-agreement = **Ferielov §21** (deadline 31 Dec); automatic post-period payout of the >4-week tranche = **§24**; feriehindring (sickness/barsel) auto-transfer ≤4 weeks = **§22** (+**§25** payout if persistent); termination payout of earned-untaken = **§26**, forskud/over-taken **modregning = §7 stk.1** (capped at outstanding pay — no clawback); untaken-not-transferred-not-paid → **§34** (Arbejdsmarkedets Feriefond); state særlige-feriedage **2½% cash godtgørelse** = Cirkulære 021-24 **§15 stk.2** + **§17**, within the **§12 stk.2** taking window (calendar-year accrual). State defers to Ferielov (Cirkulære §3).

**Design only** — settles the architecture + a phased roadmap; implementation lands in S68+ slices (D13). No execution code in S67 (optionally pure period-window value-objects + contract tests).

## Decision

**D1 — Money stays out of StatsTid; settlement emits day/hour-count lines. The verified SLS day-count finding is scoped to særlige-feriedage godtgørelse ONLY (Step-0b Codex B4).** No salary import, no in-app kroner. **Transfers (§21/§22) are balance-only — they move days to next year's `carryover_in` and emit NO payroll line.** **Payouts (§24/§26 and the §15 stk.2/§17 godtgørelse) are day-count payout lines.** The **§7 modregning is a *deduction* line whose SLS contract — including how the "capped at outstanding pay" cap is expressed/enforced — is NOT yet verified** (the research verified the SLS *day-count input* only for the særlige-feriedage godtgørelse løndele 5017/5027/5037 + ferielønregulering hours 5062; it did NOT verify a payout/transfer/deduction code or a cap flag). **Each slice's Step-0 MUST verify its mechanism's SLS wage-type input contract before that boundary is accepted** (the §7 capped-deduction is the highest-risk — a bare day-count cannot prove a monetary cap; the cap likely lives SLS-side and needs its own løndel/verification). The only in-app "amounts" remain the existing `Hours × multiplier` wage-units.

**D2 — The settlement model follows the verified §-spine.** Each boundary partitions the period-end balance into legal buckets — **transfer (§21) · auto-payout >4wk (§24) · feriehindring-transfer (§22/§25) · first-4-week forfeiture (§34) · termination payout (§26) + modregning (§7, capped) · særlige godtgørelse (§15 stk.2/§17)** — NOT the single "expiring" figure D9 shows. Recorded `absences_projection.feriedage` (ADR-032 D2) is the authoritative "used" input; `AccrualMath.EarnedToDate` crystallizes "earned-at-boundary" — both already the closed-year operand source for the D9 row, reused with no re-derivation.

**D3 — Period-close is a deterministic, idempotent BackgroundService.** A `SettlementCloseService` (Backend, the `DelegationExpiryService` 5-min-poll shape) detects due `(employee, type, closed-entitlement-year)` tuples and runs the atomic settlement (D5). **Quantity determinism (P2/P4):** the settled quantity is a pure function of an **immutable input snapshot** captured at settle time — NOT a live re-derivation. The snapshot (carried in the event payload, the S66 `EntitlementBalanceRevalued`-replacement-set template) comprises: the recorded per-absence `feriedage`, the closed-year balance (earned/used/planned/carryover_in), the dated config (annual_quota / carryover_max / reset_month / OK-version), the transfer-agreement record (D8), the termination date (D9), and the impediment status (D10) — each pinned as-of the boundary date. Replay reads the recorded snapshot, never re-derives. **Trigger idempotency:** the due-detection reads a wall-clock `CURRENT_DATE` (acceptable — a trigger, not a replayed quantity); **a poll re-evaluates the same due tuples every 5 min until an active `vacation_settlements` row exists, so poll-correctness DEPENDS on that row committing in the SAME pass that emits (D6) — else the next poll re-emits.** The due-check skips a tuple that already has an active row in ANY state {SETTLED, PENDING_REVIEW} (so a TERMINATION-settled year is skipped by the year-end poll — D5/D9 — and a PENDING_REVIEW remainder does not re-flag — D10). A missed/late poll still settles exactly once via the D5 partial-unique key. The only *new* P2/P4 surface is the boundary-date timezone — folded into the follow-up (v) business-timezone ADR (settle on the Copenhagen calendar boundary, not UTC).

**D3 clarification (S68 slice-1a — the launch-neutral go-live gate; validation fix-forward).** The automated close enforces D13's *"first boundary: 31 Dec AFTER launch; manual fallback until then"* via a configured **settlement go-live date** (`Settlement:GoLiveDate`, ISO `yyyy-MM-dd`): the `SettlementCloseService` auto-settles a closed entitlement-year ONLY if its §21/§24 boundary (31 Dec E+1) falls **strictly after** that date. Every boundary that closed before go-live is the **manual operator fallback — NOT auto-settled** — because the system has no lawful quantity source for a ferieår it never tracked (no §21 written agreement was recorded, no absences were captured; an automated "forfeit" there would be a data artifact, not an entitlement, and would violate D10's no-wrongful-forfeiture). **Unconfigured ⇒ DORMANT** (settles nothing): the launch-neutral default for this slice-1a infrastructure unit (D13 *"infrastructure groundwork, NOT a standalone shippable feature"*) — ops sets the real date at go-live, no code change. The date is a deterministic TRIGGER bound (a configured date, never wall-clock); the D3 quantity-determinism contract (settled quantity = a pure function of the captured snapshot) is unaffected. **Discovered at the S68 validation step:** the live poller, ungated, mass-flagged every seeded employee's 2020–2024 VACATION as `PENDING_REVIEW` forfeit-candidates on first host boot (forfeit=20/yr from absent historical data) — the exact opposite of launch-neutral. Fixed-forward in S68 (the gate + the dormant default + dedicated boundary tests; the close design, buckets, and state machine are unchanged).

**D3 boundary-`reset_month` integrity (S68 Step-7a B1).** The poller computes the boundary from the employee's **live** config `reset_month`, while settlement valuation reads the **dated** closed-year config — a divergence would be possible if a VACATION config could carry a `reset_month ≠ 9`. It cannot: the Danish vacation year is statutorily fixed at 1 Sep – 31 Aug (samtidighedsferie, LBK 230/2021), so **VACATION `reset_month` is enforced = 9** via a DB CHECK `entitlement_configs_vacation_reset_month` (+ the `s68-vacation-reset-month-check` legacy-upgrade ALTER) and a config-endpoint 422 guard. With `reset_month` provably uniform-9 for VACATION, the live-vs-dated read cannot diverge and the `reset_month == 1`→`31 Dec E` geometry branch is unreachable for VACATION. (CARE_DAY/SENIOR_DAY calendar-year `reset_month = 1` are unconstrained; they are not settled here.)

**D4 — Service ownership + the two cross-context exactly-once boundaries (OQ-7; Step-0b Codex B3 + Reviewer W2).** **Backend OWNS** the close: in ONE atomic tx (D5/D6) it mutates balances, writes the settlement-identity row, and emits the settlement events on `employee-{id}` — the **outbox makes the Backend state-change + event-publication atomic ONLY** (ADR-018 D3). **A NEW Payroll-context emitter CONSUMES** the settlement event to produce the export line — the existing `EventConsumerService` (in `StatsTid.Integrations.External`, an opaque-payload HTTP forwarder) is the **shape/precedent only** (poll outbox + FOR-UPDATE-SKIP-LOCKED + circuit-breaker + dead-letter), NOT the host; the settlement emitter is new, event-type-discriminated, and **needs its own business-key idempotency the existing consumer lacks**. **Payroll exactly-once (designed, not asserted):** the emitter, in ONE Payroll-side tx, writes a **consumer-checkpoint row keyed on `(settlement-identity, sequence, bucket)`** AND the export line — a redelivered event whose checkpoint exists is a no-op; the outbox redelivers until the checkpoint commits (at-least-once delivery + consumer dedup = exactly-once effect). **The `sequence` in the key is load-bearing (Codex cycle-2+3 B):** the reversal's compensating line and the superseding settlement each ride a sequence DISTINCT from the original *and from each other* (the D5 sequence-allocation invariant), so no real line is deduplicated against another; multiple `TerminationSettled` buckets are distinguished by `bucket`. Backend uniqueness prevents duplicate *closes*; this checkpoint prevents duplicate/lost *lines*. **Reversal = compensating entry, never rollback (P3):** `SettlementReversed` carries the original `(identity, sequence)`, the exact balance + export deltas, and its own idempotency key; semantically **compensate-and-leave-the-original-recorded** — it transitions the active row SETTLED → REVERSED (audited, **operator-authorized per ADR-013 — never an automatic cascade** when a settled year is later retroactively corrected); a superseding re-settlement is a NEW higher-sequence row (D5, history-preserving — NOT an in-place update; the original snapshot survives); the export side gets a compensating reversal line + the new settlement line, **each at a distinct checkpoint `(identity, sequence, bucket)`** (D5 invariant — they never share one sequence).

**D5 — ONE atomic settlement per (employee, type, closed-year), as an explicit STATE MACHINE (Step-0b Codex B2 + cycle-2 B/W; Reviewer cycle-2 W).** The boundary settles in a **single atomic pass** that partitions the closed-year remainder into ALL its legal buckets at once. The `vacation_settlements` row is keyed `(employee_id, entitlement_type, entitlement_year)` and carries `settlement_state` ∈ **{PENDING_REVIEW, SETTLED, REVERSED}**, a `sequence` (starts 1), and a `trigger` ∈ **{YEAR_END, TERMINATION}**.
- **Two keys (Codex cycle-3 B):** the **physical PK/unique is `(employee_id, entitlement_type, entitlement_year, sequence)`** (so history sequences coexist), AND a **PARTIAL unique index `(employee_id, entitlement_type, entitlement_year) WHERE settlement_state ≠ REVERSED`** enforces exactly-one-ACTIVE (the ADR-018 D8 live-row pattern). The partial index alone would permit duplicate historical sequences; the composite PK alone would not enforce single-active — both are needed.
- **PENDING_REVIEW (resolves Codex cycle-2 B + Reviewer cycle-2 W — the fail-closed-vs-poll stuck-state):** when the partition auto-resolves only part of the remainder (a §34-forfeiture / §22-feriehindring slice needs a human, D10), the pass commits the auto-resolved buckets (§21/§24/§26/godtgørelse) AND a `PENDING_REVIEW` settlement row in one tx. **The D3 due-check treats PENDING_REVIEW as NOT due** (the poll stops re-flagging it — no operational storm; the review flag is raised once, idempotent on the identity). Manual operator resolution completes the residual partition atomically and transitions PENDING_REVIEW → SETTLED.
- **Termination consumes the year-end due-check (resolves Reviewer cycle-2 W — the trigger collision):** a `TERMINATION` settlement of `(emp, type, year)` writes the single active row for that tuple; the scheduled YEAR_END close of the same year then **sees the row and skips** (the partial-unique key is the single-settle guard across BOTH triggers; `trigger` records which fired). A termination is a settlement of that year, not a second one.
- **Reversal/supersession is history-preserving + sequence-distinct (resolves Codex cycle-2 W + cycle-3 B):** a correction is **ONE atomic tx** = transition the active row SETTLED → REVERSED (it stays as history with its original D3 snapshot) AND INSERT the superseding settlement as a NEW higher-sequence row — **no intermediate-visible no-active-row window** (the D3 poll can never see the tuple un-settled mid-correction and re-derive it — the ADR-013-forbidden cascade). The active settlement is the highest-sequence non-REVERSED row. **A bare reversal (reverse-and-leave-unsettled — e.g. a year that should never have settled) is the deliberate exception: it sets a `REVERSED-without-successor` marker the D3 due-check treats as NOT-due** (mirroring PENDING_REVIEW), so the poll cannot resurrect a deliberately-unsettled year. **Transitioning the original row's *state* to REVERSED allocates NO new sequence** (it keeps its original sequence + D3 snapshot as history, and its original export line stays recorded); the compensating reversal line is instead checkpointed at a sequence/`bucket` **distinct from both the original line and the superseding line**, so the Payroll `(identity, sequence, bucket)` dedup (D4) never collapses two real lines (Codex cycle-3 B: the reversal does NOT reuse the original sequence).
- **Sequence-allocation invariant (S67 Step-7a clarification — Codex BLOCKER):** the single binding rule is **export-line uniqueness** — the original line, the compensating-reversal line, and any superseding-settlement line each own a distinct `(identity, sequence, bucket)`, so the Payroll dedup never collapses two real lines and the original is never deduped away. Transitioning a row's *state* allocates no sequence; only NEW rows do. The literal '`sequence+1`' arithmetic above is **illustrative** — taken literally it let the reversal line and the superseding line collide on one sequence; the **exact** allocation that satisfies the invariant (distinct ascending sequences, or a reversal `bucket`-variant) is a slice-Step-0b mechanism, not fixed here.
**Event vocabulary** (each `DomainEventBase` on `employee-{id}`, ADR-026 mapper + catalog row, EventTypeMap-registered): `VacationCarryoverExecuted` (§21) · `FeriehindringTransferred` (§22) · `FeriehindringPaidOut` (**§25** persistent-impediment payout) · `VacationAutoPaidOut` (§24) · `VacationForfeitedToFeriefond` (**§34**) · `SaerligeFeriedagePaidOut` (§15 stk.2/§17) · `TerminationSettled` (§26 payout + §7 modregning + §7 unearned-advance, one composite) · `SettlementManualReviewFlagged` (the D10 PENDING_REVIEW signal) · `SettlementReversed` (D4) — the §25 + §34 events were missing from the cycle-1 vocabulary (Codex cycle-2 B; they automate in slice 4 but the contract is defined now). `settlement_kind` = exactly these buckets. Each event carries the D3 immutable snapshot + the bucket day-count + the `(identity, sequence)`. **Granularity vs ADR-013:** the settlement closes year N only; seeding year N+1's `carryover_in` is a *balance write in the same tx*, NOT a boundary-spanning event.

**D6 — Balance execution: the carryover total is the DERIVED SUM of provenance-keyed transfer components, written atomically in the one settlement tx (Step-0b Codex B1+B2).** §24 is **payout, never carryover provenance** (the draft's "agreement vs §24-default" was a legal error). Carryover into year N+1 comes ONLY from **§21 (written agreement)** and **§22 (feriehindring)** — and both can transfer into the *same* next-year row, so `carryover_in` is computed as the **additive, provenance-keyed total** of the transfer components within the single atomic settlement tx (no separate-kind `SET` race), **capped at the combined statutory transfer ceiling** (Reviewer cycle-2 N — the §21 >4-week tranche + §22 ≤4-week feriehindring sum has a legal maximum; if §21+§22 would exceed it, §22 feriehindring takes precedence per its automatic statutory basis). Pre-slice-4 the §22 term is 0 (D10 — feriehindring not auto-executed), so slice 1's total = §21 only, sourced from the D8 agreement record — the ceiling check lands with slice 4. This is the **first non-zero `carryover_in` writer in project history** (the column + `UpsertAsync` exist, no production caller), via an atomic writer mirroring the S66 `ApplyRevaluationAsync` ungated upsert (NOT the booking quota guard). The active `vacation_settlements` row (D5) is written in the same tx.

**D6 clarification (S68 slice-1a — owner-ratified OQ-3): the §24-paid + §34-forfeited disposition is recorded ON the settlement row, NOT by mutating `entitlement_balances`.** ADR-032 D2 pins `used` to the recorded `absences_projection.feriedage` sum, so the close NEVER bumps `used` or "zeroes" the balance. The per-bucket disposed day-counts (`transfer_days` §21 / `payout_days` §24 / `forfeit_days` §34) live on the `vacation_settlements` row (the authoritative disposition record); **only §21 writes `entitlement_balances` (next-year `carryover_in`)**. Balance readers **special-case a SETTLED entitlement-year**: they show `remaining = 0` + the disposition from the active SETTLED-sequence's immutable snapshot (deterministic — not a live recompute), while the ferieår's **earlier monthly `saldo` is preserved** (settlement does not retroactively zero history). The settlement-row disposition is the FULL partition (§21 + §24 + §34) — a **SUPERSET** of the S66 D9 `expiring` display figure, which is ONLY the over-cap §34-candidate bucket; the slice pins this mapping (incl. PENDING_REVIEW-vs-resolved §34). The §22 feriehindring disposition term stays 0 until slice 4.

**D7 — Wage-type lines: payouts/deductions only (transfers are balance-only), via a new Payroll period-close emitter.** Per D1: §21/§22 transfers emit NO line (balance-only); §24/§26 payouts + the §15 stk.2/§17 godtgørelse emit **day-count payout lines**; §7 modregning emits a **deduction line (SLS contract + cap unverified — slice-3 Step-0, D1)**. The line is balance/period-close-derived ⇒ a **new Payroll-side emitter** (D4), NOT the rule-engine FLEX_PAYOUT seam (reuse only the `PayrollExportLine` shape + the SLS formatter). New `wage_type_mappings` rows for the settlement codes **inherit the ADR-020 versioned natural key** `(time_type, ok_version, agreement_code, position)` + `effective_from/effective_to` + the partial-unique-open-row index (not bare rows); the settlement `time_type` values are new domain entries the emitter + mappings must agree on. Day-count quantity; **SLS owns the rate** (the 2½% lives in SLS).

**D8 — §21 transfer-agreement surface: a dedicated `vacation_transfer_agreements` record (OQ-5).** `approval_periods` is too weak (WEEKLY/MONTHLY periods, time-statuses, date-range key). New table keyed `(employee_id, entitlement_year, entitlement_type)`: the agreed transfer days + the written-agreement evidence (date, actor), by the 31 Dec deadline. **Pre-launch depth: HR-recorded** (no employee+manager signing UI — the law's default is auto-payout §24, so the agreement is the exception to capture). Distinct from the §7 forskud "manager-approval-as-agreement" (ADR-030 D4 — must not conflate). The D6 carryover-write reads the §21 transferred days from this record (and, separately, §22 feriehindring days from D10); the write itself is provenance-keyed, NOT defaulted from §24.

**D9 — Termination awareness (OQ-6): a `users.employment_end_date` + the §26/§7 settlement + its audit/GDPR/auth (Step-0b Codex W5 + Reviewer RW1/N3).** New nullable `users.employment_end_date` (the S60 `employment_start_date` **HR-corrected-fact** pattern — non-bitemporal, forward-uniform on re-derivation, never in Employee/JWT/export payloads), with a **named audited mutation event `EmployeeEmploymentEndDateSet` + an ADR-026 mapper** (RBAC HROrAbove + OrgScope + If-Match is necessary but NOT sufficient for the audit contract) and a **GDPR retention/erasure** treatment (the termination date is personal data — ties to ADR-025 D3, erasure-deferred-with-D3). On set, the offboarding trigger crystallizes the final balance at the termination date and writes the active `vacation_settlements` row for `(emp, type, year-of-termination)` with `trigger = TERMINATION` (which **consumes** that year's later YEAR_END due-check — D5; a termination IS the settlement of that year, single-active-row-enforced) and emits `TerminationSettled`: **§26 payout** (day-count line) + **§7 modregning** (a *deduction* line, capped at outstanding pay — SLS contract unverified, D1) + the **§7 unearned-advance / ferie-uden-løn udligning** (same §7 mechanism — drop the ADR-030-D7 "§8-equivalent" label, no separate verified §8). **Two named slice-3 forks:** (a) **crystallization granularity** — `AccrualMath.EarnedToDate` credits whole elapsed months (clamped [0,12]), so passing `asOf=terminationDate` credits the entire termination month; whether §26 accepts whole-month or needs a day-prorated variant is a money-affecting fork (the payout is what SLS pays); (b) the **§7 deduction-line authorization** — a pay deduction may need a higher authorization bar than a payout, and triggering termination settlement is itself a privileged action. A **termination-date correction after a `TerminationSettled` emission routes through the D4 operator-authorized reverse-then-re-settle** (no silent re-settle), the same guard that protects a settled year. **Symmetrically (Codex cycle-3 B): a TERMINATION targeting a year whose YEAR_END row already exists (SETTLED or PENDING_REVIEW — only reachable via a backdated `employment_end_date` into an already-closed year) is NOT blocked by the single-active key — it routes through the SAME D4 operator-authorized reverse-(the YEAR_END row)-then-re-settle-(as TERMINATION) path** (one atomic tx, history-preserving). The normal case (mid-period termination before year-end) writes the first active row and consumes the later YEAR_END due-check; this symmetric rule covers the backdated inverse.

**D10 — Feriehindring (§22/§25) AND §34 forfeiture are NOT auto-executed pre-modeling — the close FAILS CLOSED to PENDING_REVIEW, never wrongful-forfeits (Step-0b Codex B5; cycle-2 B — the terminal state).** Auto-forfeiting (§34) an employee who is or may be feriehindret (§22) is a legal violation, not a shippable gap. **Until the impediment signal is modeled, the automated close does §21 transfer + §24/§26 payout + godtgørelse ONLY; §34 forfeiture and §22/§25 feriehindring are NOT automated.** A pass that finds an untaken-beyond-transfer forfeiture-candidate remainder commits the auto-resolved buckets + writes the settlement row in **`PENDING_REVIEW`** (the D5 terminal-until-resolved state) and emits `SettlementManualReviewFlagged` ONCE (idempotent on the identity) — **the D3 due-check treats PENDING_REVIEW as not-due, so the poll does not re-flag** (no stuck-poll / alert storm). A human operator resolves the §34-vs-§22 remainder; the manual resolution completes the residual partition via a **compare-and-set / row-version (ADR-019 If-Match) `PENDING_REVIEW → SETTLED` winner guard (Codex cycle-3 B)** — two concurrent operator completions cannot both apply the residual partition; the loser 409s rather than double-emitting the §34/§22 events. Feriehindring modeling (an impediment signal: sickness/barsel spanning the period) is slice 4; the PENDING_REVIEW manual path is the compliant default until then.

**D11 — Særlige-feriedage model correction lands in slice 2, not now (OQ-4).** Keep the ResetMonth-9/carryover-0 simplification through design; correct to the real calendar-year accrual / 1 May–30 Apr taking / §15 stk.2/§17 godtgørelse in slice 2, where the godtgørelse settlement makes the precise window meaningful (the S66 "no payoff without a payout" logic). Touches ResetMonth, seeds, AccrualMath anchoring, the D9 row.

**D12 — Settlement is GLOBAL; scope fences.** State settlement derives from one national cirkulære ⇒ **GLOBAL, no per-institution `local_configurations` override** (the ADR-025 D6 fence applied). **Fences:** no in-system kroner (D1); the 16-month ferieafholdelsesperiode is modeled only as far as the §21 (31 Dec)/§24 boundary *timing* requires (full window phases); schedule-shape stays unmodeled (ADR-032 D7); §17's 2½% ≠ §10's 2,02% (distinct, don't conflate).

**D13 — Phased roadmap (OQ-3); pre-launch settlement is MANUAL (the compliant fallback), the slices automate it (Step-0b Codex W3+W4).** The honest launch posture: **any un-automated settlement mechanism falls back to a manual operator-recorded process** — so nothing is *missing* at launch, the slices *replace manual with automated*. Each slice is marked launch-neutral with its **first-required-boundary**.
- **Slice 1 — ferie 5th-week lifecycle:** the §21 agreement record (D8) + the atomic provenance-keyed `carryover_in` write (D6) + the §24 auto-payout line (D7) + the close service (D3) + events (D5) + the Payroll exactly-once emitter (D4). The §21 carryover-write is **infrastructure groundwork WITHIN this slice, NOT a standalone shippable settlement feature** (alone it has no lawful quantity source / close trigger / audit event). *First boundary: 31 Dec after launch; manual fallback until then.*
- **Slice 2 — særlige feriedage:** the D11 model correction + the §15 stk.2/§17 godtgørelse line. *First boundary: 30 Apr after launch.*
- **Slice 3 — termination:** `employment_end_date` (D9) + §26/§7 + the granularity & auth forks. **Can occur day-1 after launch ⇒ either slice 3 ships at launch OR a manual termination-settlement process is the explicit launch fallback** (no "time-bounded" hand-wave). 
- **Slice 4 / deferred — feriehindring** (§22/§25 + the impediment signal, D10), with the §34 forfeiture automation; until then the D10 fail-closed manual review is the compliant default.

## Consequences
