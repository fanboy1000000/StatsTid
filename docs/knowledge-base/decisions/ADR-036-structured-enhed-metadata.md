# ADR-036 — Structured Enhed metadata + multi-tag membership

> **Superseded by [ADR-038](ADR-038-enhedsspor-unit-hierarchy-model.md) (S102, Enhedsspor).** The zero-authority multi-tag Enhed model (D3 + the `enheder`/`user_enheder` tables) is replaced by a typed, leader-bearing `units` hierarchy with single-unit membership (`users.unit_id`): a unit's designated leaders carry an exception-approval path over the unit's members (ADR-038 D3/D4) — Enhed is no longer pure metadata. Greenfield reseed (ADR-038 D9), so the tables are dropped, not migrated.

## Context
ADR-035 D2 flattened the org taxonomy to **MAO / Organisation / Enhed**, modelling Enhed as a flat free-text `employee_profiles.enhed_label` string (display-only, set on the ADR-022 temporal profile). The owner wants Enhed to be a real, deduplicated, **managed, multi-tag** metadata layer (a user carries several enheder — e.g. a department + a team — picked from a managed per-Organisation list), searchable, with create/rename/delete — while remaining **PURE display metadata with ZERO authority/scope/approval/payroll meaning**.

## Decision
- **D1 — Two tables.** `enheder` (`enhed_id` UUID PK, `organisation_id` FK→organizations, `name`, `deleted_at` soft-delete, `version`, `created_at`; **partial-unique `(organisation_id, lower(name)) WHERE deleted_at IS NULL`** = active dedup, deleted names recreatable). `user_enheder` (`user_id`, `enhed_id`, PK both; **both FKs `ON DELETE CASCADE`** — a junction owned by both parents; in production neither parent is ever hard-deleted [users soft-deactivate via `employment_end_date`; enheder soft-delete via `deleted_at`], so the cascade only ever fires for hard-delete cleanups). An Enhed belongs to exactly ONE **ORGANISATION**-typed org (enforced at the create endpoint, mirroring the `users.primary_org` ORGANISATION guard; a CHECK can't cross-reference `org_type`).
- **D2 — Managed-entities-first, flat, multi-tag.** An admin manages the per-Organisation enhed list; a person's tags are PICKED from that active list (no inline auto-create). Flat (untyped) — a user holds a SET of enheder. The same-Organisation invariant (each tag's `organisation_id` == the user's `primary_org_id`) is enforced at the COMMAND layer (set-tags validates against the `FOR UPDATE`-locked user's org), NOT a DB trigger — because a user transfer CLEARS the tags in the same tx.
- **D3 — ZERO authority (the load-bearing invariant).** Enhed appears in NO authority path — absent from `OrgScopeValidator`, `RoleScope.CoversOrg`, `DesignatedApproverAuthorizer` (`CanApprove`/`IsEffectiveDesignatedApprover`), `ValidateEmployeeAccessAsync`. A shared enhed grants no cross-user visibility/approval/scope. The roster/search enhed aggregation + the `?enhedId` filter sit INSIDE the users-first `accessibleOrgs` bound (keyed by `enhed_id`, not name) → no cross-org leak via a same-name enhed. Pinned by RED tests (cross-org tag reject + shared-enhed-no-CanApprove + scope-leak).
- **D4 — Event-sourced latest-wins projection** (model after `work_time_projection`/ADR-028 D1, NOT the ADR-022 temporal profile — Enhed needs no effective-dated history). Events `EnhedCreated`/`EnhedRenamed`/`EnhedDeleted` + `UserEnhederChanged`(full set, idempotent overwrite). `EnhedLabel` stays FROZEN on the `EmployeeProfile*` events + audit mappers (load-bearing in the backfill; not removed).
- **D5 — Authority & concurrency.** All enhed endpoints (`GET/POST/PUT/DELETE /api/admin/enheder` + `PUT /api/admin/users/{id}/enheder`) are **`ValidateOrgAccessAsync(LocalHR)`-floored** over the correct org (the enhed's / the person's), org-scope-contained (the S76/S91 floor; the managed-list's selectable orgs = `GetAccessibleOrgsAsync(actor, LocalHR)` — MULTI-org, not single-org). Set-tags re-validates the floor against the `FOR UPDATE`-locked org (closes the transfer-TOCTOU). Rename/delete carry the If-Match `version` predicate INSIDE the UPDATE + affected-row check (optimistic concurrency; 0-row → 404/412; no event on 0-row; 23505→409). Soft-delete + projection-filter (a soft-deleted enhed's memberships don't render; no fan-out untag write). The transfer (users PUT) CLEARS `user_enheder` in-tx, gated on the org-change predicate (a non-transfer PUT does not clear).
- **D6 — Migration.** `EnhedBackfillSeeder` reads the `enhed_label` **projection column** (DISTINCT per Organisation; NOT an event replay — the demo path never emitted `EmployeeProfileCreated.EnhedLabel`), emits `EnhedCreated` then `UserEnhederChanged`; idempotent; a no-op on the greenfield baseline (all `enhed_label` NULL). `enhed_label` kept read-only as a display fallback; the Organisation-PAGE Enhed-tree management is deferred.

## Consequences
- Enhed becomes a first-class, searchable, rename-once-updates-all metadata entity without acquiring any authority. The medarbejder-administration page gains an org-scoped Enheder management panel + a multi-select tag picker + an Enhed search filter.
- Reviews: refinement Step-4 dual-lens (2 BLOCKERs: transfer-clear atomicity + multi-org-scope) + the plan Step-0b (2 BLOCKERs: migration source/baseline-NULL + transfer-clear gating) + Step-7a ([[review-lens-complementarity]]: Codex caught 2 concurrency BLOCKERs — set-tags TOCTOU floor-escape + rename/delete lost-update — the internal lens cleared; both fixed).
- Follow-up: drop `enhed_label` once consumers fully cut over; the Organisation-page Enhed-tree (move/nesting) is the page sprint.

## S100 Amendment (2026-06-24) — Enhed becomes HIERARCHICAL (a `parent_enhed_id` tree + a derived level; STILL zero authority)
The flat Enhed gains a tree WITHIN each Organisation (owner-decided 2026-06-24; refinement Step-4 + plan Step-0b dual-lens). The hierarchy is PURE display metadata — the ZERO-authority invariant (above) is UNCHANGED and was both-lens-verified clean BY CONSTRUCTION (`parent_enhed_id` enters NO scope/approval path).
- **D7 — the tree.** `enheder.parent_enhed_id UUID NULL REFERENCES enheder(enhed_id)` (a root = NULL, directly under the Organisation; a child's parent is in the SAME Organisation — app-layer enforced, a CHECK can't cross-ref `organisation_id`). `idx_enheder_parent`. The **level is DERIVED** (the enhed depth, root = 1) — computed when the `GET /tree` assembles the forest + by the `GET /enheder` flat list (a cycle-guarded in-memory parent-chain walk); NO stored level/path column (Enhed has no roster/scope consumer — unlike the org `materialized_path`).
- **D8 — membership UNCHANGED (multi-tag).** `user_enheder` is untouched: a user carries 0..N enheder; the hierarchy is a property of the enhed ENTITIES, not of membership. Tags are an unordered set — NO ancestor inheritance, NO roll-up, NO dedupe.
- **D9 — the concurrency spine (P4).** create-child / move / delete each acquire `pg_advisory_xact_lock(hashtext('enhed-org-' || organisation_id))` (the enhed tree is wholly within one Organisation — the S95 "advisory domain = the Organisation" pattern; a DISTINCT prefix from `reporting-org-` → no false contention/deadlock), THEN run a NEW `GuardNoEnhedCycleAsync` (a recursive CTE over `parent_enhed_id`, `deleted_at IS NULL`, path-array visited-set + a depth backstop) on the held connection → concurrent enhed-tree mutations serialize (the phantom-cycle gap closed).
- **D10 — the CRUD.** Create accepts an optional `parentEnhedId` (re-read active+same-org in-tx). NEW `PUT /api/admin/enheder/{id}/move {newParentEnhedId|null}` (If-Match + version bump; under the lock + cycle CTE; `organisation_id` IMMUTABLE; in-UPDATE version predicate + 0-row → 412/404). Delete = the S97 soft-delete UNCHANGED (`deleted_at` + projection-filter; the `user_enheder` rows STAY — NOT untag) + **re-parents the child enheder UP** to the deleted enhed's parent (root → children become roots), emitting a **per-child `EnhedMoved`** in the same tx (P3 — not a silent SQL update); a leaf emits only `EnhedDeleted`.
- **D11 — events (P3).** `EnhedCreated` gains an optional `ParentEnhedId` (replay-safe: NULL=root=greenfield default; `EnhedBackfillSeeder` unchanged → roots); new `EnhedMoved`(EnhedId, OldParentEnhedId, NewParentEnhedId) + the `EventSerializer` registration; plain-outbox (consistent — only `UserEnhederChanged` is audit-registered).
- **D12 — the FE (the S99 flat-Enhed adaptation REVERSES).** The Organisation page nests enheder + re-enables Tilføj-on-Enhed (create child) + Flyt-on-Enhed (re-parent; the move picker excludes self+descendants + offers `→ Rod`) + shows the level; the Enhed-delete copy says the children lift up. The `EnhederPanel` (LocalHR) nests too. The S91 dead-button discipline applies in REVERSE — the controls are wired to real endpoints before rendering; the S99 "no Tilføj/Flyt on Enhed" vitests are inverted RED-on-old.
- **Step-7a ([[review-lens-complementarity]]):** the internal lens found the spine clean by construction; **Codex caught a BLOCKER the internal lens cleared** — `GET /enheder` dropped `parentEnhedId`/`level` from its response (the FE vitest mocked the correct shape → green while the endpoint served the wrong one — the S99 `fetchEnheder` pattern), fixed + pinned by a backend contract test. Authority is UNCHANGED — the hierarchy grants nothing (the "shared-ancestor grants no `CanApprove`/`ValidateEmployeeAccess`" RED test). Durable: SPRINT-100.md.
