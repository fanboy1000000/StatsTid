## Context
The owner redesigned org + employee administration ("Enhedsspor": the org-unit tree as the single spine, reporting derived from it) and chose **Strategy A — adopt the model fully**. The design's model: a 7-level typed ordered hierarchy (`ministeromrade → organisation → direktion → omrade → kontor → team → enhed`), each person has one structural unit, units carry explicitly-designated leaders, and reporting is presented as derived from unit membership + a primary-leader pointer. This reverses the flat-authority reform (ADR-035) and zero-authority-Enhed (ADR-036) and re-presents the reporting edge. Two consumer censuses (S102 grounding) frame the blast radius: **~100 consumers key on the Organisation home (`primary_org_id`/`org_id`)** for payroll/settlement/audit/config/JWT/scope; **19 consumers key on `materialized_path`** (4 LIKE-scoped roster reads + the delete-guard + `RoleScope.CoversOrg` [already exact-match post-S93] + the `ReparentAsync` recompute + the move event).

## Decisions

### D1 — The Organisation stays the authority anchor; the deep tree is a typed `units` table beneath it (NOT one uniform table)
Keep **`organizations`** as the top-2 levels (MAO = `ministeromrade`, Organisation = `organisation`) — the **unchanged authority / scope / config / payroll / JWT anchor**. Model the **4 intermediate levels + leaf** (`direktion`, `omrade`, `kontor`, `team`, `enhed`) as a new typed **`units`** table beneath each Organisation (evolving the S97/S100 `enheder` table: add `type`, `leader`-designation, and member/reporting semantics). The UI presents one uniform 7-level tree (`organizations` ⊕ `units`); the backend keeps the Organisation as the scope boundary.
- **Rationale:** directly satisfies the LOCKED Organisation boundary (D5); preserves the ~100 Organisation-keyed consumers via the derived home (D3); reuses the S100 hierarchical-enhed concurrency spine (advisory lock + recursive-CTE cycle guard) wholesale; bounds the migration. Semantically faithful to the 7-level model — the 2+5 split is an implementation detail invisible to the UX.
- **Rejected:** one uniform `units` table replacing `organizations` — a far larger, riskier migration (every `primary_org_id`/`org_id` FK + ~100 consumers) with no semantic gain, and it fights the locked boundary.
- `units` shape (draft): `unit_id UUID PK`, `organisation_id TEXT FK→organizations` (the owning Organisation = the derived anchor; IMMUTABLE per row), `parent_unit_id UUID NULL FK→units` (null = a top-level unit directly under the Organisation), `type TEXT` (the 5 sub-types; CHECK + the `CHILD` ordering enforced in-app), `name`, `deleted_at`, `version`, `created_at`. Level is DERIVED (depth), as in S100. The strict `CHILD` ordering (`organisation→direktion→omrade→kontor→team→enhed`) is app-enforced on create/move.

### D2 — Single-unit structural membership + a derived, denormalized Organisation home
A person's structural home becomes a single **`users.unit_id`** (their deepest unit, **nullable**) — replacing the S97 multi-enhed-tag model. **Retain `users.primary_org_id` as the always-present authority/attribution home**, set as follows (BLOCKER fix — the NULL-`unit_id` case):
- **`unit_id` NULL** (the person homes directly at the Organisation — e.g. org-level admins/directors, the existing `init.sql` Organisation-homed seed): `primary_org_id` is set **directly** to that Organisation. This is the existing behaviour, preserved.
- **`unit_id` non-null**: `primary_org_id` is the **derived Organisation ancestor** of the unit (= `units.organisation_id`), recomputed in-tx on any membership move.
So `primary_org_id` is NEVER null and the ~100 Organisation-keyed consumers (payroll attribution `EmploymentProfileResolver`, settlement filters, audit `target_org_id`/`actor_primary_org_id`, JWT `org_id` claim, config keys, org-blocking counts, the reporting authority same-Organisation check) keep reading it unchanged. Pinned by the D2 attribution test (both the NULL-`unit_id` and derived cases). `agreement_code`/`ok_version` are already per-user (bitemporal `user_agreement_codes`) — attribution there is independent of the home and needs no change.
- **No data migration (greenfield reseed — owner-decided 2026-06-28).** The system is pre-launch with test-only data, so the existing org/enhed/user/reporting rows are **dropped and recreated** into the new model via `init.sql` + the demo seed — NOT migrated. So the multi-tag→single-`unit_id` mapping is a *seed authoring* decision (each seeded user is given exactly one `unit_id`), not a migration algorithm; no "which tag wins" question arises.

### D3 — Explicitly-designated unit leaders (`leaderIds`), distinct from the reporting edge
A unit carries explicitly-designated leaders (the design's `unit.leaderIds` + the "promote to leader" checkbox). Modelled as a `unit_leaders(unit_id, user_id)` set. Decisions (BLOCKER + WARNING fixes):
- **Membership invariant (write-time, enforced):** a leader MUST be a member of the unit they lead (`user.unit_id == unit_id`). When a person's `unit_id` changes (D2 move), their `unit_leaders` rows for the OLD unit are **removed in the same tx** (the design's "synced when a person moves units"). Without this, a moved-away leader would retain D4 approval over old-unit members despite no longer being a member — closed here.
- **Write floor + lock:** `UnitLeaderDesignated`/`UnitLeaderRemoved` (D10) are **LocalHR-floored over the unit's Organisation** (`ValidateOrgAccessAsync(LocalHR)`), because designating a leader confers approval authority over that unit's members. They also **take the owning Organisation's `unit-org-` advisory** (D8) so a designation serializes with a concurrent member unit-move — closing the cycle-2 race where a designate-after-move would leave a stale `unit_leaders` row (a non-member transiently holding path-2 approval). This widens the S91 accepted lateral-secondary-principal characteristic (docs/SECURITY.md) — noted, and the S91 follow-up becomes more pressing.
- **Role coupling:** designation does NOT auto-grant a role. D4 path (2) independently requires the actor be an active **LeaderOrAbove** (the existing `IsActiveLeaderOrAboveAsync` gate), so a `unit_leaders` row for an Employee-role user grants nothing until they also hold the role — no unguarded privilege mint.
- Structural designation, distinct from the reporting edge (D4/D6). Multiple peer ("sideordnede") leaders per unit are supported.

### D4 — Approval authority (the keystone, owner-decided 2026-06-28)
`CanApprove(actor, E)` =
1. **Primary leader** — `IsEffectiveDesignatedApproverAsync(actor, E)` (the existing edge: E's primary-leader pointer → `reporting_lines` PRIMARY, vikar-aware, R3 precedence, same-Organisation re-checked). The DEFAULT path, unchanged.
2. **OR a secondary/peer leader of E's own unit** — actor ∈ `unit_leaders(E.unit_id)`, actor active LeaderOrAbove, same Organisation. The EXCEPTION path. **This makes `leaderIds` authority-bearing, BOUNDED to the unit's own direct members.**
3. **OR an active vikar** (`manager_vikar`) of E's primary leader or any of E's unit leaders (same-Organisation — D12).
4. **OR HR/Admin in scope** (`HasHrAdminScopeOverEmpOrg`, unchanged).
`DesignatedApproverAuthorizer` gains path (2)/(3) as an explicit OR after the existing edge check; the same-Organisation re-check applies to every path.
- **see == act (BLOCKER fix):** the authorizer's load-bearing contract is that the my-reports dashboard READS (`ApprovalPeriodRepository.GetPendingForDesignatedReportsAsync` / `GetByMonthForDesignatedReportsAsync`) and the action endpoints share ONE encoding so they cannot drift. The new paths (2)/(3) MUST therefore be added to those dashboard reads too (a secondary unit-leader / their vikar must SEE in their pending list exactly the employees they may ACT on). Pinned by a see==act parity test for a secondary unit-leader and for a secondary-leader's vikar.

> **D4 amendment — wired in S105 (Phase 2), ACCEPTED-AS-IMPLEMENTED.** Refinements from the S105 Step-0b dual-lens (the two convergent BLOCKERs): **(a) In-lock advisory:** the approve/reject/reopen action tx acquires the employee's `reporting-org-` THEN `unit-org-` advisory (the D8 total order) before the in-lock `CanApprove` re-eval — so a concurrent `UnitLeaderRemoved`/member-move (which serialize on `unit-org-`) cannot win a stale-authority race the edge path closed. **(b) see==act is a TWO-STAGE pipeline** (`DesignatedCandidateEmployeesCte` superset → `FilterByEffectiveApproverAsync` filter) — BOTH stages adopt the extended predicate, with the FULL floors (active + active-LeaderOrAbove + same-Org + active-vikar), centralized into ONE shared `CanApprove` helper used by every filter site (`GetTeamOverviewRosterAsync`, the allocation-breakdown gate, `ComplianceEndpoints`, the action endpoints) so no surface drifts; the my-reports reads add edge-OR-unit-leader visibility only (NOT HR/Admin scope). The medarbejder-tree TILE-count projection is scoped out (an HR aggregate, fail-closed, Phase-3). **(c) Audit:** a unit-leader approval is classified `UNIT_LEADER` / `UNIT_LEADER_VIKAR` in `approval_method` (not the misleading `ORG_SCOPE_FALLBACK`). **(d) Boundedness:** the D5 RED test uses a ≥2-level (grandparent) fixture covering BOTH action authorization AND dashboard/team-overview visibility.

### D5 — LOCKED invariant: authority scope stays at the Organisation boundary
Role-scope remains anchored at the **Organisation** (`primary_org_id`/`org_id`); the deep units (direktion→enhed) are **structure + reporting only and grant NO scope**. A parent unit's leader does **not** auto-approve a child unit's people (no deep-tree inheritance / no leader-of-leader scope). `RoleScope.CoversOrg` stays exact-Organisation-match (post-S93, unchanged). **Pinned by a RED test:** a shared deep-tree ancestor / a parent-unit leader grants no `ValidateEmployeeAccess`/approval over a descendant unit's member beyond the D4 paths. This is the guard against re-opening the S76/S85/S91 subtree-inheritance bug class.

### D6 — Stored edge RETAINED as the materialized projection (do NOT go edgeless)
`reporting_lines` (PRIMARY/ACTING, `version`, `organisation_id`) and `manager_vikar` (`organisation_id`) are **retained** as the **single canonical store** of the primary-leader edge + vikar. **Clarification (WARNING fix):** the design's "primary-leader pointer" (`person.leaderId`) IS the active PRIMARY `reporting_lines` row (`uq_reporting_line_active_primary`) — **NO new `users.primary_leader_id` column is introduced** (that would be a dual-write desync hazard). "Reporting is derived" refers to the read-side VIEWS (D7); the underlying edge is the same stored row as today, so the S74–S83 lock/revoke/`version` substrate (the drift-guarded advisory, in-lock edge re-eval, persisted revoke anchor — ADR-027 D13/D15/D18/D19) survives intact. Setting a person's primary leader writes/supersedes their PRIMARY `reporting_lines` row exactly as today. The cross-unit exception (D7) is precisely the legitimate case where the PRIMARY edge's manager is not one of the person's unit leaders — by design.
- **Rejected:** a truly edgeless model deriving approval live from `unit_id`+`leaderIds` — it would demolish the lock/revoke/version substrate and reopen the closed concurrency classes.

### D7 — Derived reporting VIEWS (presentation only, no new authority)
The design's derived views — a unit's leaders, an employee's column (grouped under their primary leader), "Refererer opad til" (leader-of-leader), the **cross-unit exception** (a person whose primary leader is outside their unit → amber + "Ret"), the **leaderless unit** (members but no leaders → amber + "Tildel leder") — are computed read-side from `unit_id` + `unit_leaders` + the primary-leader edge. They carry NO authority beyond D4.

### D8 — Concurrency regimes + the total lock order (BLOCKER + WARNING fixes)
**Units never move cross-Organisation** (`units.organisation_id` IMMUTABLE per row). So there are three operations, with one fixed total lock order to prevent deadlock.

1. **Structural unit create/move/delete (within an Organisation):** `pg_advisory_xact_lock(hashtext('unit-org-' || organisation_id))` + a recursive-CTE cycle guard over `parent_unit_id` on the held connection (the S100 spine, reused).
2. **MAO/Organisation create/move/delete:** the existing `organizations` regime (FOR-UPDATE row locks + `materialized_path` recompute, ADR-037 D5) — unchanged.
3. **A person's unit change:**
   - **Same Organisation** (new unit shares the person's current Organisation): serialize on that Organisation's `unit-org-` advisory; `primary_org_id` unchanged; update `users.unit_id` + re-sync `unit_leaders` (D3).
   - **Cross-Organisation = a TRANSFER (BLOCKER fix):** this is the existing transfer path, NOT a bare `unit_id` write. It MUST compose with the established transfer regime — acquire **BOTH** the old + new `reporting-org-` advisories via `ReportingLineRepository.AcquireRevokeTreeLocksAsync` (S83), then re-point/clear the retained **`reporting_lines` PRIMARY/ACTING + `manager_vikar` rows** whose `organisation_id` would otherwise become cross-Organisation (a cross-Organisation primary edge is forbidden — the ADR-027 / S95 same-Organisation reporting invariant, `ValidateSameOrganisationAsync` — so the transfer clears/re-anchors the edge exactly as today's same-Organisation invariant requires; today's transfer already clears enhed tags, this EXTENDS it to the reporting/vikar org-anchored rows), recompute `primary_org_id`, and write `users.unit_id`. The transfer additionally takes **both** the old + new Organisations' `unit-org-` advisories (id-sorted, mirroring the `reporting-org-` pair) so it serializes with structural unit-moves and leader-designations in either Organisation.

**Total lock order (load-bearing — advisory-before-row, ADR-027):** all `reporting-org-` advisories (id-sorted) → all `unit-org-` advisories (id-sorted) → any `users`/row `FOR UPDATE`. A single-Organisation operation takes one of each prefix; a cross-Organisation transfer takes the old+new pair of each. Acquired in this fixed prefix-then-id order in every path (structural move, leader designation, same-Org member move, cross-Org transfer), so all regimes compose without an AB/BA cycle. Pinned by a held-lock interleave test (concurrent structural-move / leader-designation vs cross-Org transfer).

### D9 — Greenfield reseed (NO migration; owner-decided 2026-06-28)
The system is in development with **test-only data** and no production history to preserve, so the org/people model is **dropped and recreated greenfield** in `init.sql` + the demo seed — there is **no backfill, no replay-across-migration, and no legacy-db-upgrade path** for this change (the runbook's non-greenfield path is explicitly out of scope here). `organizations` (MAO/Organisation) stays; the new `units` + `unit_leaders` + `users.unit_id` schema replaces `enheder`/`user_enheder`; the demo seed authors the unit tree, single-unit memberships, designated leaders, and the `reporting_lines` primary edges directly into the new shape. This eliminates the migration/replay/audit-history-preservation risk surface entirely; only **forward** event-sourcing + auditability (D10) remains in scope (priorities #3 intact for new operations).

### D10 — Events + audit-projection parity (FORWARD only)
New events: `UnitCreated` / `UnitRenamed` / `UnitMoved` / `UnitDeleted` (evolving `Enhed*`), `UnitLeaderDesignated` / `UnitLeaderRemoved`, `UserUnitChanged` (membership move). Each gets an `EventSerializer` type-map entry (DEP-003), an `IAuditProjectionMapper` + registration + an audit-projection-catalog row (ADR-026, PAT-004) so **new** operations stay auditable (P3). `audit_projection.target_org_id` resolves via the derived `primary_org_id` (D2) — FK to `organizations` preserved. Because of D9 (greenfield reseed) there is **no historical backfill or replay-across-migration to verify** — the prior `Enhed*`/`UserEnhederChanged` events simply do not exist in a fresh DB; the only determinism requirement is forward (new events replay deterministically). **(WARNING fix):** the retired `Enhed*`/`UserEnhederChanged` event types are nonetheless **KEPT name-keyed in `EventSerializer`** (the project's replay-safety convention — cheap, and a defensive guard) rather than removed. The greenfield reseed scope (D9) explicitly includes the **event store + outbox + audit_projection** (a fresh DB), so no orphaned historical events reference the retired model.

### D11 — `materialized_path` re-anchoring
The 19 `materialized_path` consumers all operate at the **Organisation level and are unchanged** (the roster LIKE reads scope by the Organisation subtree, which still exists in `organizations`; `RoleScope.CoversOrg` exact-match; `ReparentAsync` recompute). The roster reads additionally surface the deep `units` structure read-side (the Enhedsspor Struktur panel) but do not change their Organisation-scoped authority bound. No new path primitive on `units` (depth derived, S100 precedent).
- **S97/S100 enhed-consumer cutover (NOTE → Phase-1 deliverable):** dropping `enheder`/`user_enheder` (D9) breaks the live S97/S100 consumers — the `EnhederPanel` / `EnhedTagPicker`, the roster enhed-aggregation + `?enhedId` filter, and the transitional `employee_profiles.enhed_label` projection column + its readers (`AdminEndpoints`/`EmployeeProfileEndpoints`/`ApprovalPeriodRepository`/`EmployeeProfileRepository`/`Program.cs`). Phase 1 must cut these over to `units`/`unit_id` (or remove them) — enumerated here so the greenfield drop is not a silent compile/runtime break.

### D12 — Vikar (same-Organisation, render display-only)
`manager_vikar` stays authority-bearing and **same-Organisation-bound** (owner-confirmed 2026-06-28 — no loosening; S76/S83 guarantees preserved). The design's display-only "Fraværende / Vikar for …" badges are a read-side render of the active `manager_vikar` row. Parity tests: approve/reject/reopen via vikar, revoke safety, same-Organisation rejection, historical `ManagerVikar*` replay.

### D13 — Supersession disposition
ADR-035 (flat authority) **superseded** by D1/D5 (deep structure returns, but scope stays Organisation-flat — the flat-authority *scope* model is preserved, only the structure deepens). ADR-036 (zero-authority Enhed) **superseded** by D3/D4 (units carry leaders + the exception-approval path). ADR-037 (org lifecycle) **amended** by D8 (unit lifecycle generalises). ADR-008 (materialized path) **amended** by D11 (Organisation-level only; no unit path). ADR-027 (reporting/approval/vikar) **amended, not demolished** by D4/D6/D12. ADR-026 **preserve+migrate** (D10). ADR-010/014, `role_assignments`, `projects` **preserved** via the derived Organisation anchor (D2).

## Implementation test hooks (Phase 1+)
- RED: deep-tree shared-ancestor / parent-unit-leader grants nothing above the Organisation (D5).
- RED: a secondary unit-leader can approve a unit member; a leader of a sibling/parent unit cannot (D4).
- **see == act:** a secondary unit-leader (and a secondary-leader's vikar) SEES exactly the employees they may ACT on in the dashboard reads (D4).
- **unit_leaders re-sync:** moving a leader's `unit_id` removes their old-unit `unit_leaders` rows in-tx → they lose approval over old-unit members (D3).
- Held-lock interleave: concurrent structural unit-move vs cross-Org transfer compose under the fixed total lock order; the retained `reporting_lines`/`manager_vikar` substrate still serializes (D6/D8).
- Forward replay determinism + audit-projection `target_org_id` parity for the NEW events (D10) — no across-migration replay (D9 greenfield).
- Attribution: payroll/settlement/config/JWT read `primary_org_id` correctly for BOTH the NULL-`unit_id` (Organisation-homed) and the derived (deep-unit) cases (D2).
