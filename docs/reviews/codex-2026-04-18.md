## Executive Summary

- Architectural discipline has drifted in a few high-priority places: the rule-engine service now directly depends on `Infrastructure`, which contradicts the documented boundary that it should depend only on `SharedKernel`.
- The biggest correctness risk is OK-version handling. The repository has an `OkVersionResolver`, but the main write and payroll paths still trust caller-supplied `OkVersion` values instead of resolving by entry date.
- Payroll/export delivery guarantees are not at the level the docs claim. Generic wage mappings are likely invisible at runtime due to a `NULL` vs `''` mismatch, payroll delivery is still synchronous, and the external outbox consumer's locking is not safe across processing.
- Security foundations are solid in some areas (`MapInboundClaims = false`, `FindAll` for scopes, explicit scope validator), but sensitive orchestration/payroll endpoints are guarded only by `"Authenticated"`, which is too weak for retroactive recalculation and export flows.
- Static inspection shows strong rule/regression coverage and good traceability intent, but CI currently omits smoke tests and frontend tests, so some integration risks are not actually enforced.

## Top Recommendations (Prioritized)

1. Fix OK-version resolution at the write boundary and payroll boundary. Resolve `OkVersion` from each entry/absence date server-side and reject caller-supplied mismatches. Why: priorities 2-4. Effort: M. Horizon: next sprint.
2. Remove the `Infrastructure` dependency from the rule-engine project. Extract shared auth/correlation bootstrapping into a package/shared project that does not pull persistence/security infrastructure into RuleEngine. Why: priority 1. Effort: M. Horizon: next sprint.
3. Repair wage type mapping lookup semantics. Standardize generic mappings on `''` or nullable `position`, update queries accordingly, and pass actual `profile.Position` through payroll mapping paths. Why: priorities 5-6. Effort: M. Horizon: next sprint.
4. Lock down payroll and orchestrator endpoints with role-specific policies. `recalculate` should be admin-only, export endpoints should be internal/admin scoped, and orchestrator execution should not be broadly reachable by any authenticated employee token. Why: priority 7. Effort: S. Horizon: next sprint.
5. Make outbox delivery real for payroll and safe for external. Add a background payroll consumer with retry/circuit breaker, and change external polling to transactional claim-and-mark-processing semantics before send. Why: priority 5. Effort: L. Horizon: 2-3 sprints.
6. Add an automated serializer coverage test. Reflect over `DomainEventBase` descendants and fail if any are missing from `EventSerializer`. Why: priorities 3 and 8. Effort: S. Horizon: next sprint.
7. Expand CI to run smoke tests and frontend vitest. If smoke tests are too expensive per PR, gate them on main/nightly but do not leave them entirely manual. Why: priority 8. Effort: S. Horizon: next sprint.
8. Add regression tests for runtime OK-version enforcement, not just the resolver utility. Cover backend registration, weekly calculation, and payroll split/replay paths. Why: priorities 2-4 and 8. Effort: M. Horizon: next sprint.
9. Simplify governance artifacts and add drift checks. Either restore `MEMORY.md` or remove it from the workflow, and add a CI/doc validation step for referenced files plus a "quality doc vs test inventory" sanity check. Why: priorities 1 and 8. Effort: S. Horizon: 2-3 sprints.
10. Break up large endpoint files into service-layer workflows. `AdminEndpoints` and `TimeEndpoints` are carrying too much orchestration and authorization logic inline. Why: priorities 1 and 7. Effort: L. Horizon: phase 4.

## What's Working Well

- Rule/legal regression coverage is strong, especially around determinism and AC/HK/PROSA behavior separation.
- JWT claim handling is implemented with the right .NET 8 safeguards: `MapInboundClaims = false` and `FindAll` for `scopes`.
- Event store shape is fundamentally sound: append-only writes, ordered replay, actor/correlation metadata on events.
- Frontend structure is coherent: auth context, route guards, and page/module organization are clean.
- The code consistently tries to preserve traceability (`CorrelationId`, `SourceRuleId`, `SourceTimeType`), which is the right foundation to build on.
